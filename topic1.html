Data Modeling

Creating blueprints for data so it's accurate, scalable, and easy to query.

Part 1: The Theory (40 Minutes)

1. The Three Levels of Modeling

Conceptual (10 min)

High-level business view. No technical jargon. Just "What things do we care about?" (e.g., Customers, Orders).

Logical (15 min)

Defining the structure. We add attributes (Name, Date) and define relationships.

Physical (15 min)

The actual database implementation. We choose data types (INT, VARCHAR) and indexes.

2. The Core Building Blocks

  • Entities: The "nouns" (User, Product).
  • Attributes: The "adjectives" (Email, Price).
  • Relationships: The "verbs" (User buys Product).

Common Types: 1:1 (Passport), 1:N (Mother:Children), M:N (Students:Classes - Requires a junction table).

Part 2: Practical Application (50 Minutes)

Normalization: The "No Redundancy" Rule

Organization to reduce bloat. Golden Rule: Every non-key attribute must provide a fact about the key, the whole key, and nothing but the key.

1st Normal Form (1NF): No multi-valued attributes (don't put three phone numbers in one cell).

2nd Normal Form (2NF): No partial dependencies (everything in the table must depend on the primary key).

3rd Normal Form (3NF): No transitive dependencies (e.g., don't store "City" if it depends on "Zip Code" instead of the "User_ID").

Physical Implementation (Code & Schema)

1. Creating the Tables

In this script, we use SERIAL for auto-incrementing IDs and REFERENCES to enforce data integrity.

CREATE TABLE Users (
  User_ID SERIAL PRIMARY KEY,
  Username VARCHAR(50) NOT NULL,
  Email VARCHAR(100) UNIQUE
);

CREATE TABLE Orders (
  Order_ID SERIAL PRIMARY KEY,
  User_ID INT REFERENCES Users(User_ID) ON DELETE CASCADE,
  Total_Amount DECIMAL(10,2)
);

2. Visualizing the Schema (ER Diagram)

Look at how the lines (relationships) connect the IDs to understand table communication.

3. Stress Test: Querying the Model

Business Question: "Find the total amount spent by 'JohnDoe'."

Pro-Tips for Implementation:
  • Data Types: Use DECIMAL for money, never FLOAT.
  • Constraints: ON DELETE CASCADE keeps the database tidy.
  • Indexing: Add an Index to email or product columns for speed.
